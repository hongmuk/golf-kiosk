## 1. 프로젝트 진행 제안

### 1.1 프로젝트 개요

본 프로젝트는 골프 연습장, 프로샵, 골프 전문 매장 등에 설치되는 **골프공 맞춤 인쇄/각인 키오스크 소프트웨어**를 개발하는 것입니다.

고객이 키오스크 화면에서 직접 원하는 텍스트, 이미지, 캐릭터를 선택·배치하여 디자인을 완성하면, 카드 결제 후 UV 프린터로 골프공에 즉시 인쇄해 주는 **무인 주문 시스템**입니다.

**주요 요구사항:**
- Windows 기반 터치스크린 키오스크 전용 (1920×1080 FHD 고정)
- 별도 서버 없이 **단독 실행** (로컬 SQLite DB, 로컬 하드웨어 직접 제어)
- 기 구축된 하드웨어(UV 프린터, KICC 결제 단말기, 커팅기)와 연동
- 관리자 화면을 통한 매출 관리, 콘텐츠 관리, 하드웨어 모니터링
- 24시간 무인 운영을 위한 키오스크 모드 (자동시작, 키 차단, 오류 복구)

### 1.2 디자인/개발 작업 순서 및 일정

전체 개발을 **5개 단계(Phase)**로 나누어 진행합니다. 각 단계는 이전 단계의 산출물 위에 기능을 추가하는 **점진적 빌드(Incremental Build)** 방식으로, 단계마다 동작 가능한 결과물을 확인할 수 있습니다.

---

#### Phase 1. 프로젝트 기반 구축 (1주차)

소프트웨어의 뼈대를 만드는 단계입니다. 이 단계가 완료되면 빈 화면이지만 앱이 실행되고 화면 전환이 동작합니다.

| 순서 | 작업 항목 | 세부 내용 | 산출물 |
|------|-----------|-----------|--------|
| 1 | 프로젝트 스캐폴딩 | Electron + Vite + React + TypeScript 프로젝트 초기 구성, 디렉토리 구조 설계 | 빌드·실행 가능한 빈 Electron 앱 |
| 2 | 데이터베이스 설계 | SQLite 스키마 생성(5개 테이블: orders, print_logs, settings, characters, fonts), 마이그레이션, 초기 데이터 시딩 | DB 파일 자동 생성·초기화 |
| 3 | IPC 통신 브릿지 | Electron Main ↔ Renderer 간 안전한 통신 계층 구현 (contextBridge + preload), 12개 IPC 채널 정의 | 프로세스 간 데이터 송수신 |
| 4 | App Shell 구성 | React Router(HashRouter) 13개 라우트 정의, 글로벌 CSS(1920×1080 전용 레이아웃), Zustand 상태 관리 스토어 3개(app, design, order) 구성 | 화면 전환 동작하는 앱 셸 |

---

#### Phase 2. 핵심 사용자 UI (2~3주차)

고객이 실제로 사용하는 화면을 만드는 단계입니다. 특히 **디자인 편집기**가 이 프로젝트의 핵심 기능이며, 개발 비중이 가장 큽니다.

| 순서 | 작업 항목 | 세부 내용 | 산출물 |
|------|-----------|-----------|--------|
| 5 | 홈 화면 | 광고/안내 이미지 표시, 터치로 시작, 무동작 타이머(설정 가능), 관리자 진입(특정 영역 5회 터치 + 비밀번호) | 대기 → 시작 플로우 |
| 6 | 상품 선택 화면 | 골프공 인쇄 / 스티커 커팅 카드형 UI, DB에서 가격 조회, 선택 시 다음 단계 이동 | 상품 선택 완료 |
| 7 | 디자인 편집기 — 캔버스 | Konva.js 기반 원형 클리핑 캔버스(골프공 미리보기), 드래그·리사이즈 터치 조작, 레이어 패널(요소 순서 변경·삭제) | 빈 편집기 틀 |
| 8 | 디자인 편집기 — 텍스트 도구 | 텍스트 추가·편집, 온스크린 키보드(한글/영문), 폰트 선택(DB 기반), 크기 슬라이더, 12색 팔레트 | 텍스트 배치 가능 |
| 9 | 디자인 편집기 — 이미지 도구 | USB 메모리 감지(WMIC), 이미지 파일 탐색·선택, 캔버스에 이미지 배치, 크기·회전 조정 | 이미지 배치 가능 |
| 10 | 디자인 편집기 — 캐릭터 도구 | DB에서 캐릭터 목록 조회, 카테고리별 그리드 표시, 터치로 선택 시 캔버스에 배치 | 캐릭터 배치 가능 |
| 11 | 미리보기 화면 | 완성된 디자인을 큰 크기로 표시, 가격 정보, "수정하기"/"결제하기" 버튼 | 최종 확인 화면 |

---

#### Phase 3. 결제/인쇄 플로우 (4주차)

실제 돈이 오가고 제품이 만들어지는 핵심 비즈니스 로직 단계입니다.

| 순서 | 작업 항목 | 세부 내용 | 산출물 |
|------|-----------|-----------|--------|
| 12 | 결제 화면 | KICC 결제 단말기 연동, 카드 투입 안내 애니메이션, 결제 상태 표시(대기→처리→성공/실패), 재시도(3회), 타임아웃(60초) | 결제 프로세스 |
| 13 | 인쇄 진행 화면 | UV 프린터 연동, 프로그레스 바, 단계별 안내 메시지, 인쇄 상태 DB 기록(print_logs), 오류 시 관리자 호출 안내 | 인쇄 프로세스 |
| 14 | 커팅기 연동 | 스티커 상품 선택 시 커팅기 명령 전송, 상태 모니터링 | 커팅 프로세스 |
| 15 | 완료 화면 | 수령 안내, 설정된 시간 후 홈 자동 복귀, 주문·디자인 데이터 초기화 | 전체 주문 사이클 완성 |

---

#### Phase 4. 관리자 대시보드 (5주차)

매장 운영자가 사용하는 관리 화면입니다.

| 순서 | 작업 항목 | 세부 내용 | 산출물 |
|------|-----------|-----------|--------|
| 16 | 관리자 레이아웃 + 대시보드 | 사이드바 내비게이션 6개 메뉴, 오늘/주간/월간 매출 통계(SQL 집계), 하드웨어 상태 요약 | 관리자 메인 |
| 17 | 결제 내역 + 가격 설정 | 주문 목록(날짜·상태 필터), 환불 처리(KICC 취소 연동), 골프공·스티커 단가 설정 | 매출 관리 |
| 18 | 하드웨어·콘텐츠·시스템 설정 | 프린터/결제기/커팅기 상태 모니터링, 캐릭터·폰트 활성화/삭제, 비밀번호·타임아웃·장비 설정 | 운영 관리 |

#### Phase 5. 키오스크 모드 및 배포 (5주차 후반)

실제 현장 설치를 위한 마무리 단계입니다.

| 순서 | 작업 항목 | 세부 내용 | 산출물 |
|------|-----------|-----------|--------|
| 19 | 키오스크 모드 | 전체화면 고정(kiosk: true), Alt+F4·Alt+Tab 차단, 관리자 종료(Ctrl+Shift+Q), 커서 숨김, Windows 자동시작 | 무인 운영 환경 |
| 20 | Windows 패키징 | electron-builder로 NSIS 설치 파일(.exe) 생성, 설치 마법사 커스터마이징, 아이콘 적용 | 배포용 설치 파일 |
| 21 | 하드웨어 실연동 | 스텁(Stub)으로 구현된 `src/main/hardware/` 모듈을 실제 장비 SDK로 교체 (프린터·결제기·커팅기) | 현장 연동 완료 |

### 1.6 데이터베이스 설계

로컬 SQLite에 5개 테이블을 사용하며, WAL(Write-Ahead Logging) 모드로 읽기/쓰기 성능을 최적화합니다.

| 테이블 | 주요 컬럼 | 용도 |
|--------|-----------|------|
| `orders` | id, product_type, design_data(JSON), amount, payment_status, kicc_transaction_id, kicc_approval_no, print_status, created_at | 주문·결제·인쇄 상태 통합 관리 |
| `print_logs` | id, order_id(FK), status, message, created_at | 주문별 인쇄 상태 이력 추적 |
| `settings` | key(PK), value, updated_at | 시스템 설정 (가격, 비밀번호, 타임아웃, 장비 연결 정보) |
| `characters` | id, name, category, image_path, is_active, sort_order | 디자인 편집기 캐릭터 리소스 |
| `fonts` | id, name, file_path, is_active, sort_order | 디자인 편집기 폰트 리소스 |

### 1.7 하드웨어 연동 설계

모든 하드웨어 제어 모듈은 **인터페이스 추상화** 패턴으로 설계하여, 스텁(Stub) 구현과 실제 SDK 구현을 교체할 수 있도록 합니다.

| 장비 | 연결 방식 | 연동 방식 | 구현 파일 |
|------|-----------|-----------|-----------|
| UV 프린터 | USB | Windows 프린터 드라이버 또는 제조사 SDK | `src/main/hardware/printer.ts` |
| KICC 결제기 | Serial/USB | KICC DLL 호출 또는 에이전트 TCP 통신 | `src/main/hardware/payment.ts` |
| 커팅기 | USB | 제조사 SDK 또는 직접 바이너리 명령 | `src/main/hardware/cutter.ts` |
| USB 메모리 | USB | WMIC 명령(이동식 드라이브 감지) + fs(파일 탐색) | `src/main/hardware/usb-monitor.ts` |

> 현재 프로토타입에서는 모든 하드웨어 연동이 **스텁(Stub)**으로 구현되어 있습니다. 실제 장비 SDK 확보 후 `src/main/hardware/` 디렉토리 내 해당 파일만 교체하면 됩니다. IPC 인터페이스와 UI는 변경할 필요가 없습니다.
